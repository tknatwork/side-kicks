# v3.2 — Import Parser Scope

> **Status:** Scoping only. Do **not** implement in this branch.
> Implementation lands in a later session on top of this doc.

**Branch:** `feature/import-parsers-v3-2`
**Target version:** 3.2 (stacks on v3.1 format adapters)
**Author:** scope doc only — no code changes
**Last updated:** 2026-06-06

---

## 1. Why this exists

v3.1 closes the **export** side of the loop: Figma → IR → Style
Dictionary / Tokens Studio / Penpot / strict DTCG. v3.2 closes the
**import** side: any of those formats → IR → Figma Variables and
Styles. That makes the plugin genuinely bidirectional with the rest
of the ecosystem, not just with itself.

Today, the import tab assumes Figma-native JSON (or a thin DTCG
subset). Anyone pulling tokens from Style Dictionary, Tokens Studio,
or Penpot has to hand-transform first. v3.2 removes that step.

This also makes "Pull from GitHub" (v3.0.0) format-agnostic — the
v3.2 parser pipeline runs after the pull, so a repo storing
Style Dictionary tokens imports the same as a repo storing raw
Figma JSON.

---

## 2. Architecture — parser mirror of v3.1's emitter chain

The v3.1 IR (`FigmaTokenIR`) stays canonical. Parsers are the
inverse of v3.1 emitters: `JSON → parser → IR → existing Figma
import pipeline`. No new IR shape, no new Figma-side code paths.

```
   Pasted JSON / pulled from GitHub
                │
                ▼
   ┌──────────────────────────┐
   │  detectSourceFormat()    │  ← Sniffs $themes / $extensions /
   └──────────────────────────┘    legacy value-type / etc.
                │
        ┌───────┴────────┬─────────────┬──────────────┐
        ▼                ▼             ▼              ▼
  parseStyleDictionary parseTokensStudio parsePenpot parseStrictDTCG
        │                │             │              │
        └────────────────┴──────┬──────┴──────────────┘
                                ▼
                          FigmaTokenIR
                                │
                                ▼
                  Existing v2.0.0 Figma import pipeline
                  (rollback, font check, mode mapping, etc.)
```

Parsers live in the same `@side-kicks/token-adapters` package as
v3.1's emitters. Pure functions, no Figma API access, plain-Node
tested.

## 3. Source format detection

The Import tab can't ask the user every time. Detection runs on
paste, before validate.

**Heuristics (cheapest first):**

| Signal                                  | Format            |
|-----------------------------------------|-------------------|
| Top-level `$themes` array               | Tokens Studio     |
| `$extensions.com.penpot` anywhere       | Penpot            |
| `value` + `type` (no `$`) on leaves     | Style Dictionary (legacy) |
| `$value` + `$type` + multi-file index   | Style Dictionary (v4) |
| `$extensions.com.figma` on most tokens  | Figma-native (existing) |
| `$value` + `$type`, none of the above   | Strict DTCG       |

Detection returns a `{format, confidence}` tuple. UI shows the
detected format with a manual override dropdown — confidence
threshold for auto-routing is `>= 0.8`; below that we prompt.

Detection is pure-string-level (no full parse) so it stays cheap on
large pastes. Detection runs in the Web Worker alongside parsing.

## 4. Per-tool parsers

Each parser is `JSON → IR`. Pure function. Returns `{ir, warnings}`
where warnings are non-fatal (unknown fields preserved as
`$extensions.com.side-kicks-unknown`, deprecated shapes flagged,
etc.).

### 4.1 `parseStrictDTCG(json): {ir, warnings}`
Baseline. Validates against the DTCG draft schema. Reference syntax
`{path}` → `kind: 'alias'`. Unknown composite shapes → warning,
preserved verbatim in IR `$extensions`.

### 4.2 `parseStyleDictionary(json, opts): {ir, warnings}`
- Accepts both `$value`/`$type` (SD v4) and `value`/`type` (legacy).
- Multi-file inputs: caller passes a `Map<filename, json>`; parser
  merges into a single IR with mode names derived from filenames or
  `$themes`-like metadata if present.
- SD's `value` reference syntax (`{color.brand.primary.value}`)
  normalised — `.value` suffix stripped because IR refs are
  value-level by definition.

### 4.3 `parseTokensStudio(json): {ir, warnings}`
- Reads `$themes` block → IR modes (one mode per active theme).
- Token sets → IR collections.
- TS-specific shadow / typography shapes normalised to IR composite
  shape.
- Resolver order from `$metadata.tokenSetOrder` preserved so
  reference resolution matches TS semantics.

### 4.4 `parsePenpot(json): {ir, warnings}`
- DTCG-aligned input.
- `$extensions.com.penpot.modes` → IR modes.
- Penpot's string-form gradient (`linear-gradient(...)`) parsed to
  IR's structured gradient shape.

## 5. Round-trip fidelity

The IR + emitter + parser triple should be **stable under
round-trip** for the formats it owns:

```
IR ──emit──► JSON ──parse──► IR'   should satisfy   IR ≡ IR'
```

(modulo metadata like `extractedAt` timestamps).

**Golden tests in v3.2:**
- For each format `F`: load IR fixture → `emitF(ir)` → `parseF(json)`
  → assert deep-equal to original IR.
- For each format `F`: load JSON fixture → `parseF(json)` →
  `emitF(ir)` → assert deep-equal to original JSON (only for
  formats where canonical JSON is well-defined; Tokens Studio
  has key-ordering ambiguity, so we relax to structural equality).
- **Cross-format**: SD JSON → `parseSD` → IR → `emitStrictDTCG`
  → `parseStrictDTCG` → IR'. Lossy by design (SD-specific transforms
  drop); test asserts only that core token values survive.

These tests live in `packages/token-adapters/test/round-trip.spec.ts`
and run on every PR.

## 6. UI integration

Existing Import tab keeps the same layout. Two additions:

1. **Detected format badge** above the textarea, e.g.
   `Detected: Tokens Studio (high confidence)` with a dropdown to
   override.
2. **Pull from GitHub** (v3.0.0) routes through the same parser
   pipeline — pulled JSON sniffs → parses → flows into existing
   validate → preview → import.

Existing v2.0.0 features unchanged:
- Pre-import snapshot + automatic rollback.
- Font availability check.
- Library link restoration.
- Plan-limited mode selection.
- Custom Merge options.

All of these operate on the IR produced by the parser, so they
work uniformly across formats.

## 8. Open questions

### 8.1 Unknown fields — drop, preserve, or error?

Incoming JSON may have fields we don't understand (vendor
extensions, new DTCG draft fields, hand-edited custom keys).

**Options:**
1. **Drop silently.** Smallest code, loses data.
2. **Preserve in `$extensions.com.side-kicks-unknown`.** Round-trip
   safe even for unrecognised fields. Slight schema bloat.
3. **Error.** Strictest, worst UX.

**Recommendation:** (2) for non-leaf metadata, (1) for leaf
unknown keys with a warning. Errors only on broken structure
(invalid JSON, missing required `$value`/`value`).

### 8.2 Multi-file Style Dictionary input

Style Dictionary's natural shape is one file per mode/theme. Our
import textarea is single-paste.

**Options:**
1. Require user to merge files into one JSON object keyed by
   filename before paste.
2. Accept a `.zip` upload of multiple files (Figma plugins can read
   `File` inputs in `ui.html`).
3. Accept a JSON manifest that lists file paths + contents.

**Recommendation:** (2). Aligns with the existing Import tab's
upload affordance and keeps the SD use case first-class.

### 8.3 Conflict resolution when importing into an existing Figma file

If a parsed IR conflicts with existing Figma variables (same
collection + same path, different value), what happens?

This is the same Custom Merge problem v2.0.0 already solved for
Figma-native imports — the IR plugs into the existing Custom Merge
pipeline. No new UI needed, but we should regression-test it with
parsed-from-SD input to make sure the merge dialog behaves the same.

### 8.4 Detection on ambiguous JSON

`{"foo": {"$value": "#fff", "$type": "color"}}` is valid as both
strict DTCG and Tokens Studio. What does detection return?

**Options:**
1. Default to **Strict DTCG** when no TS-specific markers present.
   TS users get a dropdown override.
2. Surface "Ambiguous — pick one" and block auto-parse.

**Recommendation:** (1). DTCG is the safer default because both
emitters can round-trip strict DTCG; only TS-specific input loses
fidelity if mistakenly parsed as DTCG.

## 9. Out of scope for v3.2

- **Inline conversion in the GitHub-stored format.** v3.2 parses on
  import — it does **not** rewrite the source repo to a different
  format. Format migration is a manual step.
- **SD transform replay.** SD's transforms (e.g. `color/css`,
  `size/px`) produce derived outputs; we ignore them. Users wanting
  SD-transformed output run SD on the JSON we emit.
- **Live two-way sync.** Watching a repo for changes and
  auto-importing isn't viable inside a Figma plugin (no
  background workers, no notifications). Manual Pull only.
- **Non-token assets.** Component schemas, page exports, image
  assets — all outside this scope.
- **Generic JSON Schema validator.** We validate against the four
  format-specific shapes we own. Custom schemas are user error
  surfaced as warnings, not gracefully handled.

## 10. Acceptance criteria for v3.2.0

- [ ] `detectSourceFormat()` correctly classifies all fixtures in
      `packages/token-adapters/test/fixtures/` with confidence
      `>= 0.8`.
- [ ] Strict DTCG + Style Dictionary parsers ship.
- [ ] Round-trip tests green for both emitters + parsers.
- [ ] Cross-format test (SD → IR → DTCG → IR') passes the
