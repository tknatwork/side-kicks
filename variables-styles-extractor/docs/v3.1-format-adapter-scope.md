## 1. Why this exists

The plugin currently has a `figma | w3c` format switch
(`src/code.ts:238`, `src/code.ts:812`). The `w3c` exporter is
deliberately thin:

- Type map is 4 entries: `color`, `float → number`, `string`, `boolean`
  (`src/code.ts:817`). No `typography`, `shadow`, `gradient`,
  `transition`, `cubicBezier`, `dimension` composites.
- Color is hex-only — no `color()` P3 syntax, no `oklch`.
- No reference validation; aliases pass through as raw `{path}` strings.
- Multi-mode collections fall back to mode subgroups, which is **not**
  DTCG-canonical — each downstream tool has its own theme model
  (Tokens Studio `$themes`, Style Dictionary file-per-mode, Penpot
  `$extensions`).

W3C DTCG is the nominal common substrate for Style Dictionary, Tokens
Studio for Figma, and Penpot — but each one's **parser** diverges in
practice:

| Tool                  | DTCG flavour quirks                                                                                                 |
|-----------------------|---------------------------------------------------------------------------------------------------------------------|
| Style Dictionary v4+  | Accepts `$value`/`$type` but transforms expect specific shapes for shadow/typography. Legacy `value`/`type` fallback still common. |
| Tokens Studio for Figma | Adds `$themes` block, custom token sets, its own ref resolver order. "DTCG export" still differs from strict DTCG. |
| Penpot                | DTCG-aligned but uses `$extensions` for mode/theme metadata; gradient shape differs.                                  |

So the work is real. The question is **architecture**, not whether to
build it.

## 2. Architecture — IR + per-tool emitters

**Reject:** N×M direct exporters (`figma → SD`, `figma → TS`,
`figma → Penpot`, `figma → DTCG`). Maintenance cost grows linearly
with target count; each new format touches extraction code.

**Adopt:** Figma → **canonical Internal Representation (IR)** → tiny
per-tool emitter.

```
                    ┌──────────────────────┐
   Figma Variables  │  extractFigmaToIR()  │
   + Styles ──────► │  (one pass, full     │ ──► FigmaTokenIR
                    │   fidelity)          │
                    └──────────────────────┘
                                                       │
                                                       ▼
   ┌─────────────────────┬────────────────┬────────────────┬──────────────────┐
   │ emitStyleDictionary │ emitTokensStudio│ emitPenpot     │ emitStrictDTCG   │
   └─────────────────────┴────────────────┴────────────────┴──────────────────┘
                                                       │
                                                       ▼
                                              JSON to clipboard /
                                              file / GitHub (v3.0.0)
```

Each emitter is ~50–100 lines because the IR has already normalised
modes, references, composites, and metadata.

## 3. The Internal Representation (FigmaTokenIR)

The IR is **not a publish format** — it's an internal schema designed
for fidelity and fast conversion. Sketch:

```ts
interface FigmaTokenIR {
  schemaVersion: '1.0';
  source: {
    file: string;            // Figma file key
    extractedAt: string;     // ISO timestamp
    pluginVersion: string;
  };
  collections: IRCollection[];
  styles: IRStyle[];          // Effects, typography, paint styles
  references: IRReferenceGraph; // Resolved alias graph for cycle detection
}

interface IRCollection {
  id: string;                 // Figma collection id
  name: string;               // Original Figma name
  modes: IRMode[];
  variables: IRVariable[];
}

interface IRVariable {
  id: string;
  path: string[];             // e.g. ['color', 'brand', 'primary']
  type: 'color' | 'number' | 'string' | 'boolean'
      | 'dimension' | 'typography' | 'shadow' | 'gradient'
      | 'cubicBezier' | 'duration';
  values: Record<ModeId, IRValue>; // Value per mode
  scopes?: FigmaScope[];
  description?: string;
  hiddenFromPublishing?: boolean;
}

type IRValue =
  | { kind: 'literal'; value: string | number | boolean }
  | { kind: 'color';   value: IRColor }       // sRGB + P3 + oklch
  | { kind: 'alias';   ref: string[] }        // Path-based ref, not raw {x.y}
  | { kind: 'composite'; value: IRComposite } // Typography, shadow, gradient
```

Key invariants:
- **References are paths, not strings.** Emitters re-stringify them in
  the target syntax (`{path.to.token}`, `$path.to.token`, etc.).
- **Modes are first-class.** Emitters decide how to project them
  (TS → `$themes`, SD → file-per-mode, Penpot → `$extensions`).
- **Colors carry all gamuts.** Emitter picks hex / `color()` / `oklch`
  based on target capability.
- **Composites are normalised.** Typography is always `{fontFamily,
  fontSize, fontWeight, lineHeight, letterSpacing}` — emitter renames
  keys per target.

## 4. Per-tool emitters

Each emitter is a single file, pure function `IR → JSON`. No
side effects, no Figma API access.

### 4.1 `emitStrictDTCG(ir): DTCGTokenTree`
The baseline. Strict W3C DTCG with `$value`/`$type`. Single-mode
output; if `ir` has multiple modes, this emitter requires a `mode`
parameter and emits the selected mode. References stringified as
`{dot.path}`.

### 4.2 `emitStyleDictionary(ir, opts): SDTokenTree`
- `opts.legacyShape?: boolean` — emit `value`/`type` instead of
  `$value`/`$type` for SD < v4.
- Composites flattened into SD-expected shapes (e.g.
  `boxShadow.offsetX` not `shadow.x`).
- Multi-mode → multi-file (one tree per mode + an index).

### 4.3 `emitTokensStudio(ir): TSExport`
- Token sets per collection.
- `$themes` block synthesised from modes.
- Refs in TS's `{path}` syntax (same as DTCG, but TS resolver is
  order-sensitive — emitter sorts sets topologically).
- TS-specific metadata in each token (`description`, `$type`).

### 4.4 `emitPenpot(ir): PenpotTokenTree`
- DTCG-aligned with `$extensions.com.penpot` for mode/theme metadata.
- Gradient shape uses Penpot's `linear-gradient(...)` string form, not
  DTCG's structured `gradient` type (which Penpot doesn't fully
  support yet).

## 5. Packaging

**`@side-kicks/token-adapters`** — a standalone TS package living at
`variables-styles-extractor/packages/token-adapters/` (or sibling
folder; decide during implementation).

Dual-loaded:

1. **Bundled into `ui.html`** as an inline IIFE so the Figma plugin
   has zero runtime deps. Build step uses `esbuild` to produce a
   single JS string that gets concatenated into `ui.html`.
2. **Published to npm** as `@side-kicks/token-adapters` so headless
   use is possible — e.g. a GitHub Action that regens Style
   Dictionary output from the IR JSON written by v3.0.0's GitHub
   sync, without ever opening Figma.

Package shape:

```
packages/token-adapters/
  src/
    ir/
      types.ts
      validate.ts          // Cycle detection, ref resolution
    emitters/
      strict-dtcg.ts
      style-dictionary.ts
      tokens-studio.ts
      penpot.ts
    index.ts
  test/
    fixtures/              // IR JSON snapshots from real Figma files
    emitters.spec.ts       // Run each emitter, diff vs golden output
  package.json
  tsconfig.json
```

Tests run in plain Node — **no Figma sandbox**, no QuickJS. CI gate
catches emitter regressions before plugin re-publish.

## 6. UI / plugin integration

Export tab gains a **target picker**:

- Figma (canonical — existing behaviour)
- Strict W3C DTCG
- Style Dictionary
- Tokens Studio
- Penpot

Picker drives which emitter runs. Hands off to:

- Clipboard / download (existing).
- **Push to GitHub** (v3.0.0) — target picker chooses what gets
  written. The GitHub sync path becomes format-agnostic.

Import tab gains a **source format detector** that sniffs incoming
JSON ($themes → TS, `value`/`type` → SD legacy, `$extensions.com.penpot`
→ Penpot, otherwise → DTCG/Figma) and routes to the inverse
parser (parsers ship in v3.2 — see §10).

## 7. Efficiency at scale

Why this design holds up for "1k+ variables × 5 modes × 6
collections" design systems:

- **Single extraction pass.** `extractFigmaToIR()` walks
  `figma.variables.getLocalVariablesAsync()` once. Emitters never
  touch the Figma API.
- **Web Worker reuse.** v2.0.0 already runs JSON parse/stringify in a
  Web Worker. Emitters run there too — main thread stays free.
- **Streaming-friendly.** IR is array-of-collections; emitters can
  process collections one at a time and concatenate output. Memory
  ceiling stays low even on huge libraries.
- **Caching.** v2.0.0's LRU cache (10 entries / 60s) keys on a hash
  of the IR. Re-emit to a different target is cache-hit on IR
  extraction; only the emitter pass runs.
- **No Figma re-review on emitter changes.** Bug in the Style
  Dictionary emitter? Patch the npm package, ship a plugin point
  release with the updated inline bundle. Figma Community review is
  fast for patch versions; the architectural surface area didn't
  change.

## 9. Open questions

### 9.1 Composite token canonical shape

DTCG `typography`, `shadow`, `gradient` shapes are still being
finalised by the working group. Each tool implemented an early draft
slightly differently.

**Options:**
1. Pick the **latest DTCG draft** as IR canonical, emit-time convert
   to per-tool legacy shapes.
2. Pick **Tokens Studio's shape** (most widely deployed in design
   teams) as IR canonical.
3. Define an **IR-only shape** that's neither, with explicit
   per-tool conversion.

**Recommendation:** (3). The IR is internal — let it carry the
fields Figma actually exposes, and let emitters do the translation.
Keeps the IR stable as DTCG evolves.

### 9.2 Reference resolution timing

Refs can be resolved at extraction time (IR carries resolved values
+ alias path) or at emit time (each emitter walks the ref graph).

**Options:**
1. Resolve at extraction, IR carries both `kind: 'alias'` and the
   computed final value.
2. Resolve at emit, IR carries only `kind: 'alias'` paths.

**Recommendation:** (1) — costs ~5% more memory but lets emitters
flatten refs when the target tool doesn't support aliases (rare
edge case), and makes cycle detection a one-time check.

### 9.3 Mode → theme projection

Each target tool has a different mental model:
- SD: theme = directory of files.
- TS: theme = entry in `$themes` array.
- Penpot: theme = `$extensions.com.penpot.modes`.
- DTCG: no native theme concept; pick one mode per emit.

**Options:**
1. Emitters own the projection; IR just carries `modes: ModeId[]`
   per variable.
2. IR includes a `themes: ThemeProjection[]` hint, computed once.

**Recommendation:** (1). Theme semantics differ enough that a shared
hint would just be ignored by half the emitters. Per-emitter
projection is clearer.

### 9.4 Strict-DTCG single-mode requirement

Strict DTCG has no theme/mode concept. If the user has multi-mode
collections and picks "Strict DTCG", what do we emit?

**Options:**
1. Require a mode parameter; emit only that mode.
2. Emit per-mode files like SD does.
3. Pick mode 0 silently and log a warning.

**Recommendation:** (1) — explicit choice surfaces in UI.

## 10. Out of scope for v3.1

- **Import parsers** (`SD → IR`, `TS → IR`, `Penpot → IR`,
  `DTCG → IR`). Lands in v3.2. v3.1 is export-side only; users
  importing into Figma still go through the existing pipeline.
- **Round-trip fidelity tests.** Need import parsers first.
- **Custom user transforms.** SD has rich transform pipelines; we
  don't replicate them. Users wanting custom transforms run SD on
  the JSON we emit.
- **CSS / SCSS / iOS / Android output.** Those are SD's job — emit
  to SD, let SD do the rest.
- **GitHub Action templates.** Document the npm package's usage,
  but don't ship Actions.
- **Wide-gamut color round-trip with Figma.** Figma's color API
  still returns sRGB; P3/oklch in IR comes from Figma's wide-gamut
  flag where present, but full P3 round-trip is gated on Figma API
  changes we don't control.

## 12. Sequencing with v3.0.0

- v3.0.0 (PR #19) ships GitHub sync with the **existing** `figma`
  and `w3c` formats. No adapter dependency.
- v3.1.0 ships the IR + first two emitters. GitHub sync gains the
  target picker.
- v3.1.1 / v3.1.2 add Tokens Studio and Penpot emitters.
- v3.2 adds import parsers, enabling round-trip from any supported
  format back into Figma.

Each version is independently shippable. v3.1 does not block on
v3.0.0 implementation finishing — both branches can land in parallel
and merge in either order, because the GitHub sync touches
`code.js`/`ui.html` and the adapter package is a new directory.
